<objective>
Close the three independently-confirmed BLOCKER gaps (BL-01, BL-02, BL-03) plus the
WARNING gap (BL-04) identified by the Phase 1 verification report
(`.planning/phases/01-thermocline-py-foundations/01-VERIFICATION.md`,
`status: gaps_found`, `score: 3/5 must-haves verified`).

These gaps share a single root cause: the brine adapter and `Verifier` were tested only
with **synthetic flat-dict envelopes** and a **single shared in-memory keystore**, masking
three real production paths:

1. **BL-01** — Cross-role verification is structurally impossible. `BrineProvider.public_key()`
   reads the **private seed** from the keystore and derives the verify key from it. A
   verifier-only role (e.g., a forge) cannot verify any signature without holding the
   signer's private seed — direct violation of the spec's "the public key is the node
   identity — share it freely" guarantee. Phase 3 SC3 (cross-role round-trip
   Photophore → pi-forge) cannot land until this is fixed.

2. **BL-02** — `Verifier.verify` reads `envelope.get('key_scheme')` from the **top level**,
   but every real envelope shape (Task, TaskResult, Job, JobResult) nests `key_scheme` under
   `dispatch_signature.key_scheme` (for `task`/`job`) or `receipt_signature.key_scheme`
   (for `task_result`/`job_result`). Every legitimate signed envelope is rejected with
   `SchemeError("declared key scheme None does not match signature scheme 'brine'")`.
   Phase 1 SC2 (round-trip via thermocline-py only) is broken on real envelopes.

3. **BL-03** — `BrineProvider.__init__` probes `type(backend).__name__` for the substrings
   `'fail'` / `'null'`. The real production classes are `keyring.backends.fail.Keyring` and
   `keyring.backends.null.Keyring` — both class-named **`'Keyring'`**. The substring check
   passes silently for them, so the adapter falls open in headless / no-D-Bus environments
   (Linux servers without a desktop session). IDENT-05 fails open in production.

4. **BL-04** (WARNING, opportunistic close) — `BrineProvider.generate()` calls
   `set_password` unconditionally; re-running setup destroys the prior signing identity.
   Key rotation is supposed to be a deliberate, audited operation. (Not strictly an SC1-5
   gate, but a foreseeable data-loss path; closing it here while we are in the file.)

Closure path is one focused plan with six discrete tasks (a Wave-0 conftest fixture, then
one task per gap, then a CHANGELOG update). Tasks 2-5 all touch the same file
(`thermocline/python/src/thermocline/identity.py`) so they execute sequentially within a
single wave to avoid merge conflicts. Test files are modified per task; one new test file
per closure proves the closure independently.

real-envelope test) because Task 3's regression tests call `register_public_key` which is

```python
"""Shared pytest fixtures for the thermocline-py test suite.

This file is consumed by every test file in ``tests/`` (pytest auto-discovers
``conftest.py``). It provides the ``brine_in_memory_keyring`` fixture used by
the BL-01 / BL-02 / BL-03 / BL-04 closure tests added in Plan 01-04.

Coexistence note (Plan 01-04 / W7 closure):
The pre-existing module-scoped ``fake_keyring`` fixture inside
``tests/test_identity_brine_roundtrip.py`` is intentionally preserved as-is.
That fixture uses a ``MagicMock`` backend and is wired ONLY for the existing
13 brine round-trip tests. The new ``brine_in_memory_keyring`` fixture below
uses a REAL ``keyring.backend.KeyringBackend`` subclass so the BL-03 isinstance
probe sees a live, non-fail/non-null class. The two fixtures are NOT
interchangeable (the MagicMock cannot satisfy isinstance checks against
real keyring backend classes); they coexist intentionally.
"""
from __future__ import annotations

from typing import Iterator

import keyring
import pytest
from keyring.backend import KeyringBackend


class _InMemoryKeyringBackend(KeyringBackend):
    """Real ``KeyringBackend`` subclass backed by a per-instance dict.

    Class name is deliberately NOT ``Keyring`` so it is unambiguously distinct
    from ``keyring.backends.fail.Keyring`` and ``keyring.backends.null.Keyring``
    (both of which the BL-03 isinstance probe rejects).
    """

    priority: float = 100  # type: ignore[assignment]

    def __init__(self) -> None:
        self._store: dict[tuple[str, str], str] = {}

    def set_password(self, service: str, username: str, password: str) -> None:
        self._store[(service, username)] = password

    def get_password(self, service: str, username: str) -> str | None:
        return self._store.get((service, username))

    def delete_password(self, service: str, username: str) -> None:
        self._store.pop((service, username), None)


@pytest.fixture
def brine_in_memory_keyring() -> Iterator[_InMemoryKeyringBackend]:
    """Install a real in-memory ``KeyringBackend`` for the duration of one test.

    Yields the backend instance so a test may introspect ``backend._store`` if
    it needs to assert no key bytes leak elsewhere. Restores the
    previously-installed backend on teardown so test isolation is preserved.
    """
    previous = keyring.get_keyring()
    backend = _InMemoryKeyringBackend()
    keyring.set_keyring(backend)
    try:
        yield backend
    finally:
        keyring.set_keyring(previous)
```

```python
    Lookup order (BL-01 closure — load-bearing):

    1. Public-key store — ``keyring.get_password(self._keyring_service,
       _PUBKEY_PREFIX + identity)``. Populated by :meth:`register_public_key`.

       other nodes without ever holding their seeds.

    if pub_hex is not None:
        try:

        except ValueError as exc:
```

```python
def rotate(self, *, identity: str) -> None:
    """Replace the seed for ``identity`` with a freshly-generated one.

    BL-04: this is the ONLY documented path that overwrites an existing seed.
    The previous behaviour — ``generate`` silently overwriting — was a
    foreseeable data-loss path (re-running a setup script destroys the prior
    signing identity).

    Post-condition: the seed stored at ``(self._keyring_service, identity)``
    is different from the seed that was there before this call. Any
    ``register_public_key`` entry for the same identity is independent (it
    lives under a different keystore key) and is NOT touched by this call.

    Raises
    ------
    IdentityError
        Code ``IDENTITY_NOT_FOUND`` if no seed exists for ``identity`` to
        rotate. (Use :meth:`generate` first.)
    """
    existing = keyring.get_password(self._keyring_service, identity)
    if existing is None:
        raise IdentityError(
            f"cannot rotate identity {identity!r}: no seed exists "
            "(use generate() first)",
            code="IDENTITY_NOT_FOUND",
        )
    new_signing_key = nacl.signing.SigningKey.generate()
    keyring.set_password(
        self._keyring_service, identity, new_signing_key.encode().hex()
    )
    del new_signing_key
```
