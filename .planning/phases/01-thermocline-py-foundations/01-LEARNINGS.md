# Phase 1 Learnings: thermocline-py-foundations

## Decisions

### D-01 — Receipt private constructor via module-private sentinel token
`Receipt.__init__` requires a keyword-only `_token: _ReceiptConstructorToken` parameter with no default. The token type is module-private, and a runtime identity check (`_token is _RECEIPT_TOKEN`) rejects foreign instances. Both `mypy --strict` (compile-time) and `TypeError` (runtime) gates fire on direct construction.

**Rationale:** Locks the verify-only construction invariant under both static and dynamic checks. External callers cannot type-check the token because the type itself is private — pure runtime checks alone would let typed callers slip through; pure static checks alone would let `# type: ignore` callers slip through. Belt-and-suspenders is required for a privacy primitive.
**Source:** 01-CONTEXT.md, 01-03-SUMMARY.md

---

### D-02 — JSON Schema artifacts generated from Pydantic models with CI drift gate
Schemas live committed under `thermocline/schema/*.schema.json` (Draft 2020-12), generated by `python -m thermocline.scripts.build_schemas`. CI runs `--check` mode that fails on any drift between Pydantic-generated schemas and the committed artifacts.

**Rationale:** The schemas are the cross-language contract; manual editing diverges from the Pydantic source-of-truth. The `--check` gate ensures the JSON artifacts and Python types never desync, which is required because future Rust/TypeScript ports will validate against the JSON files, not the Python.
**Source:** 01-CONTEXT.md, 01-01-SUMMARY.md

---

### D-03 — `Sensitive[T]` is a Python-language repr-only redaction wrapper
`Sensitive[T]` redacts `__repr__` (returning `'<redacted>'`) but NOT serialization or hashing. The wrapper is type-discipline scaffolding for the privacy boundary, not a cryptographic primitive.

**Rationale:** Operational privacy primitives (canonical JSON, signing, audit chain) carry the actual security guarantees. `Sensitive[T]` exists to make accidental leakage via `print(envelope)` or log statements visually obvious during development; conflating it with crypto would create false confidence.
**Source:** 01-CONTEXT.md, 01-01-SUMMARY.md

---

### D-04 — Three-level YAML conformance manifest schema with phase tagging
`thermocline/conformance/MANIFEST.yaml` (top), `valid/MANIFEST.yaml` and `invalid/MANIFEST.yaml` (mid), per-fixture entries (bottom). Every entry carries a `phase:` field declaring which phase wires it behaviorally. AT-C5/C6 wired in Phase 1; AT-C1/C2/C3 carry `phase: 2` for Photophore wiring; AT-C4 wired behaviorally during Phase 1 gap closure.

**Rationale:** The phase tag turns the conformance corpus into a forward-promising contract — Phase 2 and Phase 3 can grep for their tagged fixtures and know their wiring obligations. Without it, "structural-only" fixtures look identical to behavioral ones and rot silently.
**Source:** 01-CONTEXT.md, 01-03-SUMMARY.md

### Top-level `key_scheme` fallback retained as the single sanctioned spec deviation
`Verifier._declared_scheme` reads the canonical nested location (`dispatch_signature.key_scheme` for `task`/`job`; `receipt_signature.key_scheme` for `task_result`/`job_result`; `None` for `task_error`/`job_error`), but falls back to top-level `envelope.get("key_scheme")` for typeless envelopes and envelopes whose nested block is absent or empty.

**Rationale:** Strictly following the spec would have churned five existing `_minimal_envelope` tests in `test_identity_brine_roundtrip.py` (synthetic flat-dicts predating the gap closure). Documented as an explicit fallback rule rather than a hidden quirk.
**Source:** 01-04-SUMMARY.md

---

### `BrineProvider` separates public-key store from seed store via `_PUBKEY_PREFIX` namespace
Public keys for foreign identities are stored in the SAME platform keystore as seeds, but under a `pubkey:` prefix. `register_public_key(*, identity, verify_key)` is the only documented foreign-key write path. `public_key()` consults the pubkey store FIRST, then falls back to seed-derived for self-signing.

**Rationale:** Cross-role verification (a verifier-only node verifying foreign signatures) was structurally impossible while `public_key()` derived from the seed. The lookup-order invariant (pubkey store first) is load-bearing — it lets `rotate()` replace the seed without touching the registered public key for the same identity.
**Source:** 01-04-SUMMARY.md

---

### `BrineProvider.generate()` refuses clobber; explicit `rotate()` is the only overwrite path
`generate(identity=X)` raises `IdentityError(code='IDENTITY_ALREADY_EXISTS')` on second call. New `rotate(identity=X)` replaces an existing seed and leaves any pubkey-store entry for the same identity untouched.

**Rationale:** Re-running a setup script would silently destroy the prior signing identity and orphan all signatures it had produced. Key rotation is a deliberate, audited operation by definition; making it look identical to first-time generation creates a foreseeable data-loss path.
**Source:** 01-04-SUMMARY.md

---

### Keystore-availability probe uses `isinstance` against real production classes, not name substring
`BrineProvider.__init__` checks `isinstance(backend, (keyring.backends.fail.Keyring, keyring.backends.null.Keyring))` to reject non-functional backends. The earlier substring check on `type(backend).__name__` for `'fail'` / `'null'` is removed.

**Rationale:** Both production fallback classes are named `Keyring` — substring `'fail'` and `'null'` match neither. The substring heuristic silently fell open in headless Linux / no-D-Bus deployments, which is exactly the IDENT-05 failure mode the spec requires be rejected.
**Source:** 01-04-SUMMARY.md

---

### `signature_hash` recipe: `blake2b(canonical_bytes + signature.bytes_, digest_size=32).hex()`
The Receipt's `signature_hash` field hashes the concatenation of the canonical-JSON bytes and the Ed25519 signature bytes. Pinned by a known-good vector documented in 01-03-SUMMARY.md for cross-language port alignment.

**Rationale:** Phase 4's conformance harness needs a regression vector for any Rust/TypeScript port to match. blake2b is stdlib; signature-as-tail keeps the recipe linear and easy to reproduce.
**Source:** 01-03-SUMMARY.md

---

### Plan 01-04 task ordering: Wave 0 conftest → BL-01 → BL-02 → BL-03 → BL-04 → CHANGELOG
The plan-checker took 3 iterations to converge on this ordering. Task 2 (BL-01) lands BEFORE Task 3 (BL-02) because BL-02's tests call `register_public_key`, which lands in BL-01. BL-01's headline cross-role test uses a DUAL-FORM envelope (top-level + nested `key_scheme`) so it survives the OLD top-level `Verifier.verify` lookup still in place at BL-01 land-time.

**Rationale:** Captured here so a future maintainer doesn't try to "simplify" by reordering. The dual-form envelope is the load-bearing trick that makes each task individually green at land-time, rather than requiring the full chain to land atomically.
**Source:** 01-04-SUMMARY.md, HANDOFF.json

### AT-C4 fixture wired with `Signature(scheme=KeyScheme.PGP, ...)` not `KeyScheme.BRINE`
The fixture's `_signature_actual_scheme` metadata is `'pgp'`. The behavioral test must construct a `Signature(scheme=KeyScheme.PGP, ...)` for the `declared='brine' != actual='pgp'` mismatch path to fire. An earlier draft used `KeyScheme.BRINE` and would have hit the no-provider-registered branch — a passing test for the wrong reason.

**Rationale:** Plan-checker B4 correction. AT-C4's whole purpose is the scheme-mismatch path; using `KeyScheme.BRINE` would have given green-test theater while leaving the actual surface uncovered.
**Source:** 01-04-SUMMARY.md

## Lessons

### `Sensitive[T]` needed a `__get_pydantic_json_schema__` hook to schema-serialize correctly
Pydantic v2's automatic schema generation for parameterized generic types didn't produce the desired schema for `Sensitive[bytes]` until an explicit `__get_pydantic_json_schema__` classmethod was added that resolves the `T` parameter and emits the inner type's schema directly. The base type for the JSON Schema is the inner T (e.g., `bytes` → `string` with `format: byte`); the `Sensitive` wrapper is invisible to JSON consumers.

**Context:** Discovered during Plan 01-01 Task 3 (schema generation). Pydantic v2 generic-type schema introspection is shallower than the v1 path; advanced types must hook explicitly.
**Source:** 01-01-SUMMARY.md

---

### RFC 8785 normalizes `1.0 == 1` to the same canonical bytes
The Plan 02 specification claimed `canonicalize(1.0) != canonicalize(1)` based on a misreading of JCS. RFC 8785 / ECMAScript number canonicalization actually emits `1` for both — `1.0` is stripped of its trailing zero. The plan was corrected during execution.

**Context:** Discovered during Plan 01-02 implementation. The correction is regression-pinned in `tests/test_canonical.py` so future readers don't re-introduce the misreading.
**Source:** 01-02-SUMMARY.md

---

### Pydantic v2 `ConfigDict(extra='forbid', frozen=True)` is the load-bearing combination
The combination — strict extra rejection (catches AT-C6 scenarios) plus immutability (catches accidental in-place mutation after canonicalization) — is exactly what the privacy boundary needs. Either one alone leaves a gap.

**Context:** Plan 01-01 envelope type design. Without `extra='forbid'`, AT-C6 (extra field) fixtures would silently parse; without `frozen=True`, code could mutate an envelope after signing and corrupt the signature.
**Source:** 01-01-SUMMARY.md

---

### Static lints (AST-based) outperform regex for "no `json.dumps` outside allowlist"
`thermocline.scripts.check_no_json_dumps` walks the AST and inspects `Call` nodes by attribute name. A regex-based version would have produced false positives on string literals containing `'json.dumps'` (e.g., docstrings, comments).

**Context:** Plan 01-02 enforcement of THERMO-04 (single canonical-JSON path). Phase 1 has no docstrings mentioning `json.dumps` yet, but Phase 2 likely will when documenting the prohibition.
**Source:** 01-02-SUMMARY.md

---

### Plan 03's frontmatter named test files that the executor never committed
Tasks 1-2 of Plan 03 listed `tests/test_identity_brine_roundtrip.py` and `tests/test_identity_keystore_required.py` in the plan frontmatter, but the executor's commits did not include them. The implementation in `identity.py` was correct; the proving tests were absent. IDENT-02 and IDENT-05 were behaviorally uncovered until Task 3 backfilled them.

**Context:** Plan-frontmatter list of files is not enforceable as a checklist by the GREEN gate — the gate runs the test suite and trusts whatever tests exist. A future workflow could grep the frontmatter against the actual diff.
**Source:** 01-03-SUMMARY.md

---

### Executor agent committed directly on `main` from inside a worktree (not on a feature branch)
The Plan 02 executor was launched with `isolation="worktree"` but ended up committing directly to `main`. The worktree branch was never created or merged — the execution happened transparently on `main` from inside what was supposed to be an isolated checkout.

**Context:** Discovered post-execution while reviewing git log. The harness contract for worktree isolation was violated; downstream impact was zero (commits were correct) but the safety property was lost. Plan 04's executor exhibited the same behavior — six commits linear on `main`.
**Source:** 01-02-SUMMARY.md, 01-04-SUMMARY.md

---

### Mypy `--strict` rejects `dict.get()` returns when the dict is `dict[str, Any]`
`_declared_scheme` returns `str | None`, but `envelope.get("key_scheme")` returns `Any` (because the dict is `dict[str, Any]`), and mypy refuses to narrow `Any` to `str | None` automatically. Explicit `str(scheme)` casts at every return site are required, not `# type: ignore[no-any-return]`.

**Context:** Discovered during Plan 04 Task 3 (BL-02 implementation). The plan's pseudo-diff used `# type: ignore` comments; the executor swapped them for explicit casts because mypy flagged some as unused.
**Source:** 01-04-SUMMARY.md

---

### Synthetic flat-dict test envelopes silently masked four production lookup-path bugs
The Plan 03 test suite used synthetic flat-dict envelopes (top-level `key_scheme: "brine"`) instead of the real nested envelope shapes. All four BL-* defects (cross-role impossibility, nested-lookup absence, broken keystore probe, silent generate clobber) were invisible until verification ran the production code paths against real conformance fixtures.

**Context:** Discovered in the gaps_found verification report. Synthetic shortcuts in tests are technical debt with a privacy-primitive cost — they look like coverage but don't exercise the production paths.
**Source:** 01-04-SUMMARY.md, 01-VERIFICATION.md

---

### Substring-on-class-name keystore probes silently fall open in production
The original `BrineProvider` rejected backends whose `type(backend).__name__` contained `"fail"` or `"null"`. Both production classes — `keyring.backends.fail.Keyring` and `keyring.backends.null.Keyring` — are named `'Keyring'`, so the probe matched neither. Tests passed because the synthetic look-alikes were named `FailKeyring` and `NullKeyring`.

**Context:** BL-03 root cause analysis. Substring heuristics on framework-internal names are anti-patterns for security gates — `isinstance` against the real classes is the only correct probe.
**Source:** 01-04-SUMMARY.md

---

### Plan-checker iteration is a strict net positive: 11 → 3 → 0 issues across 3 rounds
Plan 01-04 went through 3 plan-checker rounds (11 issues found → 3 → 0). Each iteration tightened ordering, tightened test design, tightened lookup-table completeness — none was a restart. The convergence cost was minutes; the avoided in-execution defect cost would have been hours.

**Context:** First time the plan-checker → planner revision loop was exercised end-to-end on this project. The pattern of "convergent iteration is cheap, restart is expensive" is now empirically supported.
**Source:** Plan 01-04 execution log; 01-04-SUMMARY.md "Issues Encountered: None"

## Patterns

### Module-private sentinel for verify-only constructor
A frozen dataclass exposes a constructor that requires a module-private sentinel object as a keyword-only parameter. The sentinel's TYPE is module-private (so external typed callers can't satisfy mypy) AND a runtime identity check rejects foreign instances (so untyped or `# type: ignore` callers still fail). Used for `Receipt`.

**When to use:** Cryptographic value types whose construction is meant to be a witness — i.e., the act of constructing implies an upstream verification path was traversed. Both static and runtime gates are required for true belt-and-suspenders.
**Source:** 01-CONTEXT.md, 01-03-SUMMARY.md

---

### `@runtime_checkable` Protocol for cross-implementation duck-typing
`IdentityProvider` is a `typing.Protocol` decorated `@runtime_checkable`. Concrete adapters (`BrineProvider`, future PGP/X.509 adapters) need not subclass; passing the duck-type check is sufficient.

**When to use:** Cross-implementation interfaces where the suite must accept third-party adapters that can't import the suite's protocol class. Standard ABC inheritance would couple all adapters to the suite's class hierarchy.
**Source:** 01-03-SUMMARY.md

---

### Frozen dataclass + redacted `__repr__` for cryptographic value types
`Signature(bytes_=..., scheme=..., issuer=...)` is `@dataclass(frozen=True, slots=True)` with a custom `__repr__` that emits `Signature(scheme='brine', issuer='alice', bytes_=<redacted 64 bytes>)`. The raw signature material never appears in logs or REPL output.

**When to use:** Any value type holding cryptographic material (signatures, public keys, future MACs). Frozen prevents accidental mutation; slots minimizes memory; redacted repr prevents accidental leakage via `print()` or log statements.
**Source:** 01-03-SUMMARY.md

---

### Type-routed lookup table with documented fallback
`Verifier._declared_scheme(envelope)` switches on `envelope.get("type")` against an exhaustive table of all five envelope discriminator values, then falls back to a documented top-level read for typeless / empty-block cases. The table is exhaustive; the fallback rule is named and tested.

**When to use:** Any cross-shape lookup that must be exhaustive over a closed set of discriminator values, but also tolerate legacy / synthetic shapes during a transition. The fallback must be explicit (not implicit) and exercised by an explicit test.
**Source:** 01-04-SUMMARY.md

---

### `isinstance` against framework production classes, not name substring
Backend rejection probes import the actual fallback classes (`keyring.backends.fail.Keyring`, `keyring.backends.null.Keyring`) and check `isinstance` against them. Class-identity is reliable across framework versions; class-name strings are not.

**When to use:** Any security gate that must reject specific framework-internal types. Substring on `__name__` is an anti-pattern; framework authors are free to rename.
**Source:** 01-04-SUMMARY.md

---

### Three-level YAML conformance manifest with per-fixture phase tags
`MANIFEST.yaml` at the corpus root, `valid/MANIFEST.yaml` and `invalid/MANIFEST.yaml` mid-level, per-fixture entries (with `surface`, `description`, `expect_error_code`, `phase`, `notes`) at the leaf. Every entry's `phase:` field declares which phase wires the fixture behaviorally.

**When to use:** Cross-language conformance corpora that ship structurally before all phases can wire them behaviorally. The phase tag prevents structural-only fixtures from looking identical to wired ones.
**Source:** 01-03-SUMMARY.md

---

### Plan-checker → planner revision loop with bounded iteration cap (3)
Plan-checker spawns subagent that critiques the plan, returns a structured issue list. Planner subagent revises in place. Loop until checker reports 0 issues OR cap reached. Plan 01-04 converged in 3 rounds (11 → 3 → 0).

**When to use:** Any non-trivial plan (more than 2 tasks or any cross-task ordering constraint). The iteration cost is small; in-execution defect costs are large. Stall detection (no net issue reduction across two consecutive iterations) should trigger a hard stop and human escalation.
**Source:** Plan 01-04 execution log

---

### Real-fixture-from-disk regression tests over synthetic shortcuts
Behavioral regression tests load JSON fixtures from `thermocline/conformance/{valid,invalid}/` via `Path(__file__).resolve().parents[3]` rather than constructing in-test dicts. This closes the synthetic-test loophole.

**When to use:** Any test that crosses an envelope boundary (parsing, signing, verifying). Synthetic in-test envelopes test the test-author's assumptions; real fixtures test the production path.
**Source:** 01-04-SUMMARY.md

---

### Schema generation pipeline with CI drift gate
Pydantic models are the source of truth; `build_schemas` script regenerates committed JSON Schema artifacts; `--check` mode in CI fails on drift between generated and committed schemas.

**When to use:** Cross-language contracts where the canonical type definitions live in one language but consumers are language-agnostic. The drift gate is what makes the JSON artifacts trustworthy as the contract.
**Source:** 01-CONTEXT.md, 01-01-SUMMARY.md

---

### AST-based static lint (not regex) for forbidden API calls
`check_no_json_dumps` walks the AST and inspects `Call` nodes by attribute name. False positives from string literals (`"json.dumps"` in docstrings) are structurally impossible.

**When to use:** Any forbidden-import or forbidden-call gate where the forbidden token is likely to appear in legitimate documentation. Regex is unsafe; AST walks at the syntactic level the lint actually cares about.
**Source:** 01-02-SUMMARY.md

---

### Hypothesis property-based tests for crypto invariants
Plan 01-02 ships 5 hypothesis property tests over canonicalization (round-trip stability, key-order independence, whitespace independence, etc.). Hypothesis explores the input space rather than asserting specific examples.

**When to use:** Any invariant on a function that must hold for all inputs in a class — canonical-JSON stability, signature-verify round-trip, audit-chain hash determinism, shadow-ID uniqueness. Examples test points; properties test regions.
**Source:** 01-02-SUMMARY.md

## Surprises

### Pydantic v2 generic schema introspection requires explicit `__get_pydantic_json_schema__` hook
`Sensitive[bytes]` did not generate the inner-type schema automatically. The Plan 01-01 executor expected Pydantic v2 to introspect generic parameters automatically (as v1's deeper introspection allowed). It does not.

**Impact:** ~30 minutes of debugging during Plan 01-01 Task 3. The hook is now in place and documented in `Sensitive[T]`'s class docstring as a deviation from "automatic" generic-schema generation.
**Source:** 01-01-SUMMARY.md

---

### RFC 8785 normalizes `1.0` and `1` to identical canonical bytes
Plan 01-02's spec text claimed they produced different bytes; RFC 8785 actually canonicalizes both to `"1"` (trailing-zero strip). The plan was corrected during execution and the test was inverted.

**Impact:** Brief mid-Plan-02 spec correction; would have been a cross-language signature mismatch in Phase 4 if shipped uncorrected. Now regression-pinned.
**Source:** 01-02-SUMMARY.md

---

### Plan 03 executor silently skipped two named test files from the plan frontmatter
`test_identity_brine_roundtrip.py` and `test_identity_keystore_required.py` were named in Plan 03's frontmatter but the executor's commits did not include them. The implementation in `identity.py` was correct; behavioral coverage for IDENT-02 and IDENT-05 was missing until Task 3 backfilled.

**Impact:** Required Plan 03 Task 3 to absorb 14 extra tests across the two missing files. The behavioral coverage gap was the seed of the BL-* defects discovered later — no test exercised the production lookup paths.
**Source:** 01-03-SUMMARY.md

---

### Plan 03 verification flipped from PASS to gaps_found (3/5) on second run
The first verification reported PASS; the second (more rigorous) verification using independent harness scripts that loaded real conformance fixtures from disk found 4 BL-* gaps. The synthetic test envelopes had masked all four production defects.

**Impact:** Phase 1 required a fifth plan (01-04 gap closure) to actually close. Re-verification after gap closure restored 5/5. Now treated as the canonical example for "synthetic-test loophole" in this project.
**Source:** 01-VERIFICATION.md, 01-04-SUMMARY.md

### Both keyring fallback classes are named `'Keyring'` — substring probe matched neither
`type(keyring.backends.fail.Keyring()).__name__ == 'Keyring'`. Same for `null.Keyring`. The substring `'fail'` / `'null'` heuristic the original code used was a tautological no-op.

**Impact:** BL-03. The IDENT-05 spec requirement (refuse to start without working keystore) was structurally violated in production; only synthetic test classes (which were named to contain the substring) made the test suite pass.
**Source:** 01-04-SUMMARY.md

---

### Worktree-isolated executors committed directly to `main`
Both Plan 02 and Plan 04 executors were launched with `isolation="worktree"` but their commits landed linearly on `main` rather than on isolated branches. The merge / branch-creation step expected by the worktree contract did not happen.

**Impact:** No data damage in either case (commits were correct), but the safety property of worktree isolation was lost. A future buggy executor could have corrupted `main` directly. Worth a follow-up to either fix the worktree contract or stop claiming isolation.
**Source:** 01-02-SUMMARY.md, 01-04-SUMMARY.md, .continue-here.md

### Plan 01-04 executed in 12 minutes (722 seconds) for 6 tasks across 4 BL-* gaps
The pre-execution estimate (implicit in plan size) suggested 30-45 minutes. Actual: 722s. The plan-checker iteration ate ~20 minutes upfront; the execution itself was tight because every task was crisp.

**Impact:** Confirms "front-load planning, execute fast" is a workable rhythm for gap-closure scope. The pattern likely doesn't generalize to greenfield phases — Plan 01-01 (greenfield) took longer per task.
**Source:** 01-04-SUMMARY.md frontmatter `duration: 12min`

### CHANGELOG W10 acceptance check contradicted the file's actual heading style
Plan 01-04's W10 closure check required `grep -c '^### 0.3.1' thermocline/CHANGELOG.md == 1`, but the existing CHANGELOG used `## v0.3.1` (level-2 with `v` prefix). Both checks could not literally pass simultaneously.

**Impact:** Auto-fixed during Task 6 by honoring the load-bearing intent ("no duplicate v0.3.1 heading") rather than the literal acceptance regex. Lesson for future planners: acceptance grep patterns should be derived from the actual file state, not invented from intuition.
**Source:** 01-04-SUMMARY.md "Deviations from Plan"
