<decisions_made>

Plan 01-04 gap-closure decisions (full rationale in `.planning/HANDOFF.json` and the v0.3.1 CHANGELOG entry):

- **BL-01** — `BrineProvider` separates the public-key store from the seed store via a `_PUBKEY_PREFIX = "pubkey:"` namespace. `register_public_key(*, identity, verify_key)` is the only documented foreign-key path. `public_key()` consults the pubkey store FIRST, then falls back to seed-derived for single-node self-signing. This restores the "the public key is the node identity — share it freely" guarantee and unblocks Phase 3 SC3 (cross-role round-trip Photophore → pi-forge).

- **BL-02** — `Verifier._declared_scheme` is a `@staticmethod` with type-routed lookup over all five envelope discriminator values (`task`/`job` → `dispatch_signature.key_scheme`; `task_result`/`job_result` → `receipt_signature.key_scheme`; `task_error`/`job_error` → `None` since error envelopes are unsigned by spec). Top-level `envelope.get('key_scheme')` is preserved as a tolerated fallback for synthetic flat-dict tests AND typed envelopes whose nested block is absent or empty — this preserves the 5 existing `_minimal_envelope` tests in `test_identity_brine_roundtrip.py` without churning them.

- **BL-03** — `isinstance(backend, (keyring.backends.fail.Keyring, keyring.backends.null.Keyring))` replaces the substring-on-`__name__` heuristic. Both production classes are named `'Keyring'`, so the substring check fell open silently. Tests in `test_identity_keystore_required.py` now import the real classes (replaced the synthetic `FailKeyring`/`NullKeyring` look-alikes whose `__name__` did contain the substring — false coverage signal).

- **BL-04** — `BrineProvider.generate(identity=X)` raises `IdentityError(code='IDENTITY_ALREADY_EXISTS')` on second call. New `BrineProvider.rotate(identity=X)` is the only documented path that overwrites an existing seed. Pubkey store entries are namespaced separately and are NOT touched by `rotate` (orthogonality preserved).

- **Plan ordering** — Wave 0 (conftest) → Wave 1-5 (BL-01 → BL-02 → BL-03 → BL-04 → CHANGELOG). Task 2 (BL-01) lands BEFORE Task 3 (BL-02) because Task 3's tests call `register_public_key`. Task 2's headline cross-role test uses a DUAL-FORM envelope (top-level + nested `key_scheme: 'brine'`) so it survives the OLD top-level `Verifier.verify` lookup that's still in place at Task 2 land-time. The plan-checker took 3 iterations to converge on this ordering — captured here so a future maintainer doesn't try to "simplify" by reordering.
</decisions_made>
